# devbase 统一分析：双模态定位

> 整合 `competitive-roadmap-table-a.md`（人类赛道）与 `ai-infrastructure-analysis.md`（AI 赛道）
> 核心结论：**两份分析都正确，devbase 是同时服务人类和 AI 的"双模态工具"**

---

## 一、为什么两份分析都成立

### 第一份分析的价值（人类赛道）

```
用户场景：开发者坐在终端前，按 `devbase` 打开 TUI
          看到 20 个仓库的状态，按 `s` 批量安全同步
          
价值：真实、当下、可感知
用户：人类开发者（当前主力）
竞品：lazygit, gitui, desktop, gws
```

**客观事实**：
- 人类现在就是 devbase 的用户
- TUI 的流畅度直接影响留存
- lazygit 在多仓库场景确实是盲区
- 安全同步策略（SyncPolicy）是人类用户的刚需

### 第二份分析的价值（AI 赛道）

```
用户场景：Claude Code 通过 MCP 调用 devbase
          获取本地代码库的全景状态
          
价值：未来、差异化、壁垒
用户：AI Agent（未来增长）
竞品：5ire, dify, claude-code-rust
```

**客观事实**：
- AI 确实无法识别 GUI
- MCP Server 确实是 AI 访问本地代码库的最佳方式
- 结构化 SQLite 注册表是 AI 的"原生食物"
- 当前做这个方向的几乎没人

### 两份分析不矛盾，是时间轴的两个点

```
时间轴：

Now          6 months          12 months         2 years
  │              │                 │                │
  ▼              ▼                 ▼                ▼
人类为主  →  人类+AI并行  →  AI调用增长  →  AI主导，人类看板
  │              │                 │                │
  │  TUI 是核心   │  TUI + MCP      │  MCP 是核心     │  自然语言交互
  │  竞品 lazygit │  双模态差异化    │  竞品 5ire      │  竞品无
  │              │                 │                │
  └──────────────┴─────────────────┴────────────────┘
                    devbase 的演进路径
```

│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │

│         │                 │                    │            │
│         └─────────────────┼────────────────────┘            │
│                           ▼                                 │
│              ┌─────────────────────┐                        │

│              │  │ • summaries    │  │                        │
│              │  │ • module_graph │  │                        │
│              │  └───────────────┘  │                        │
│              └──────────┬──────────┘                        │
│                         │                                   │

**关键设计**：
- **统一数据层**（SQLite）是核心，人类和 AI 读同一份数据
- **TUI** 是人类的"只读视图 + 有限操作"
- **MCP Server** 是 AI 的"完整 API"
- **CLI** 是两者的交集（人类打字，AI 调用）

## 三、竞品的双维度重评估

### 3.1 人类维度评分

| 竞品 | 人类体验 | 多仓库支持 | TUI 质量 | 安全策略 | 人类赛道威胁 |
|:-----|:-------:|:---------:|:-------:|:-------:|:----------:|
| lazygit | ⭐⭐⭐⭐⭐ | ❌ | ⭐⭐⭐⭐⭐ | ❌ | 🔴 高 |
| gitui | ⭐⭐⭐⭐ | ❌ | ⭐⭐⭐⭐ | ❌ | 🟡 中 |
| desktop | ⭐⭐⭐⭐ | ❌ | N/A(GUI) | ❌ | 🟡 中 |
| gws | ⭐⭐ | ✅(基础) | ❌ | ❌ | 🟢 低 |
| **devbase** | ⭐⭐⭐ | ✅ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | — |

**人类赛道结论**：
- devbase **不试图击败 lazygit** 的单仓库体验
- devbase 的人类价值是**"多仓库仪表盘 + 安全策略"**
- lazygit/gitui 是**伙伴**（通过启动器集成），不是敌人

### 3.2 AI 维度评分

| 竞品 | AI 可用性 | 结构化数据 | MCP 支持 | 代码库语义 | AI 赛道威胁 |
|:-----|:--------:|:---------:|:-------:|:---------:|:----------:|
| 5ire | ⭐⭐⭐ | ⭐⭐(向量) | Client | ⭐⭐ | 🔴 高 |
| claude-code-rust | ⭐⭐⭐ | ⭐⭐(Skill) | ❌ | ⭐⭐⭐ | 🟡 中 |
| codex | ⭐⭐ | ❌ | ❌ | ⭐⭐ | 🟡 中 |
| dify | ⭐⭐ | ⭐⭐ | ❌ | ⭐ | 🟢 低 |
| lazygit | ⭐ | ❌ | ❌ | ❌ | 🟢 低(人类工具) |
| **devbase** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | Server | ⭐⭐⭐ | — |

**AI 赛道结论**：
- devbase 在 AI 可用性和结构化数据上是**独一档**
- 5ire 是最接近的竞品，但知识库类型不同（文档 vs 代码库原生）
- **短期无人竞争**，但 5ire/claude-code-rust 可能快速跟进

### 3.3 双维度综合定位

```
                    AI 能力强
                         │
                         │
     5ire ●              │              ● dify
    (知识库+             │             (LLM平台)
     MCP Client)         │
                         │
                         │
    claude-code ●        │              ● codex
    (AI编码)             │             (AI编码CLI)
                         │
                         │        ● devbase
    ─────────────────────┼─────────────────────
    人类体验差           │           人类体验好
                         │
    ● gws                │              ● desktop
                         │
                         │        ● lazygit
                         │        ● gitui
                         │
                         │
                    AI 能力弱
```

**devbase 的位置**：
- 右上角象限的**唯一玩家**
- 人类体验不如 lazygit（承认），但**AI 能力碾压**
- AI 能力不如 5ire（在通用知识上），但**代码库语义深度碾压**

## 四、统一战略：双轮驱动

### 4.1 两个轮子必须同时转

```
        ┌─────────────┐
        │   devbase   │
        └──────┬──────┘
               │
       ┌───────┴───────┐
       ▼               ▼
  ┌─────────┐    ┌─────────┐
  │ 人类轮   │    │ AI 轮    │
  │ TUI/CLI │    │ MCP/API │
  └────┬────┘    └────┬────┘
       │               │
       ▼               ▼
  ┌─────────┐    ┌─────────┐
  │ 当前收入 │    │ 未来壁垒 │
  │ 用户留存 │    │ 差异化   │
  └─────────┘    └─────────┘
```

**只转人类轮** → 沦为 better gws，天花板低  
**只转 AI 轮** → 没有用户基础，MCP 无人调用  
**双轮同时转** → 人类用户带来数据，数据训练 AI 能力，AI 能力吸引新场景

### 4.2 双轮的具体分工

| 维度 | 人类轮（当下） | AI 轮（未来） |
|------|--------------|--------------|
| **核心目标** | 用户留存、口碑传播 | 生态位建立、标准制定 |
| **关键功能** | TUI 流畅度、SyncPolicy、标签 | MCP Server、结构化查询、代码语义 |
| **竞品对标** | lazygit（多仓库盲区） | 5ire（代码库语义深度） |
| **成功指标** | DAU、仓库注册数 | MCP 调用次数、AI 集成数 |
| **投入比例** | 70%（当前） | 30%（当前）→ 50%（未来） |

### 4.3 整合后的蚕食路线

```
Phase 1: 人类基础（Now - Month 3）
  ├─ TUI 体验打磨（启动器、排序、过滤）
  ├─ SyncPolicy 强化（--dry-run、防误操作）
  └─ 多仓库场景验证（统计用户仓库数分布）
  
Phase 2: 双模态建立（Month 3 - Month 9）
  ├─ MCP Server tool 扩展（查询类 tool）
  ├─ 注册表 Schema 升级（code_metrics、module_graph）
  ├─ TUI 新增"AI 建议"面板（展示 AI 对当前仓库的分析）
  └─ 与 1-2 个 AI 助手建立集成（Claude Code / 5ire）
  
Phase 3: AI 主导（Month 9 - Year 2）
  ├─ 自然语言查询（人类在 TUI 中输入自然语言，AI 翻译为 SQL）
  ├─ 智能同步建议（AI 判断最优 SyncPolicy）
  ├─ 代码语义搜索（tree-sitter 索引）
  └─ 成为 AI 助手的"代码库理解"标准接口
```

## 五、竞品的正确关系图谱

### 5.1 人类维度关系

```
devbase ──互补──→ lazygit（单仓库精细操作）
   │
   ├──互补──→ gitui（同上）
   │
   ├──互补──→ desktop（GUI 偏好用户）
   │
   └──吃掉──→ gws（功能全面超越）
```

### 5.2 AI 维度关系

```
devbase ──竞合──→ 5ire（Client vs Server，知识库类型互补）
   │
   ├──上下游──→ claude-code-rust（Claude 调用 devbase 获取上下文）
   │
   ├──上下游──→ codex（Codex 调用 devbase）
   │
   ├──无关──→ lazygit/gitui/desktop（AI 用不了）
   │
   └──友军──→ syncthing-rust（代码层 + 文件层双同步）
```

### 5.3 双维度综合关系

| 竞品 | 人类维度 | AI 维度 | 总体关系 |
|------|---------|--------|---------|
| lazygit | 🔴 竞争（单仓库体验更好） | 🟢 无关（AI 用不了） | **互补** |
| gitui | 🟡 竞争（部分重叠） | 🟢 无关 | **互补** |
| desktop | 🟡 竞争（GUI vs TUI） | 🟢 无关 | **无关** |
| gws | 🔴 竞争（功能全面超越） | 🟢 无关 | **吃掉** |
| 5ire | 🟢 无关（不同形态） | 🔴 竞争（知识库赛道） | **竞合** |
| claude-code-rust | 🟢 无关 | 🟡 竞争（AI 编码上下文） | **上下游** |
| codex | 🟢 无关 | 🟡 竞争 | **上下游** |
| syncthing-rust | 🟢 无关 | 🟢 无关 | **友军** |

## 六、关键风险（双维度视角）

| 风险 | 人类维度影响 | AI 维度影响 | 综合评估 |
|------|:----------:|:----------:|:-------:|
| 多仓库是伪需求 | 🔴 高（核心功能失效） | 🟢 低（AI 仍可用单仓库知识库） | 🟡 中 |
| TUI 体验差于 lazygit | 🔴 高（人类用户流失） | 🟢 低（AI 不走 TUI） | 🟡 中 |
| MCP 生态不成熟 | 🟢 低 | 🔴 高（AI 用户增长慢） | 🟡 中 |
| 5ire 加入代码库语义 | 🟢 低 | 🔴 高（直接竞争） | 🔴 高 |
| 资源分散（双轮） | 🟡 中 | 🟡 中 | 🔴 高（最大风险） |

**最大风险**：双轮驱动需要双倍资源，单人/小团队可能两头都做不好。

**应对**：
- 明确阶段重心：Phase 1 人类轮 70%，Phase 2 逐步平衡，Phase 3 AI 轮主导
- 不做完美 TUI（不追 lazygit 的单仓库体验），做到"够用即可"
- 不做完美 AI（不追 5ire 的通用知识库），做到"代码库语义最深"

## 七、修正后的执行清单

### 立即执行（Week 1-2）

1. **人类轮**：`Enter` 启动 gitui/lazygit（2 天）
2. **人类轮**：`--dry-run` 强化 + 危险策略警告（1 天）
3. **AI 轮**：MCP tool 返回 JSON 格式（1 天）
4. **双轮**：README 重写 — 明确"双模态：人类 TUI + AI 知识库"定位（半天）

### 短期执行（Month 1-3）

1. **人类轮**：TUI 过滤/搜索优化
2. **AI 轮**：新增查询类 MCP tool（`query_repos_by_language`, `query_repos_by_tag`）
3. **双轮**：注册表增加 `code_metrics` 表（代码行数、语言比例）
4. **验证**：统计用户仓库数量分布，验证多仓库需求真实性

### 中期执行（Month 3-9）

1. **AI 轮**：tree-sitter 代码语义索引
2. **AI 轮**：与 Claude Code 建立 MCP 集成案例
3. **人类轮**：TUI 新增"AI 洞察"面板（展示仓库健康趋势、建议）
4. **双轮**：模块依赖图可视化

## 八、一句话总结

> **devbase 不是 Git TUI，也不是纯 AI 基础设施——它是同时服务人类和 AI 的"代码库双模态操作系统"。**
>
> 人类通过 TUI 看仪表盘，AI 通过 MCP 查知识库，两者共享同一个 SQLite 注册表。
>
> 短期靠人类体验活下来，长期靠 AI 壁垒活得好。
